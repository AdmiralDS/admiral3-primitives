# Contributing

В этом документе зафиксированы обязательные правила внесения изменений в репозиторий.

## Коммиты

Для сообщений коммитов используется `Conventional Commits`:

https://www.conventionalcommits.org/ru/v1.0.0/

### Формат сообщения

Базовый формат:

```text
type(scope): short description
```

Примеры:

```text
feat(ExampleComponent): add storybook playground template
fix(testing): correct vitest typings for jest-dom matchers
docs(contributing): describe pull request preparation flow
chore(ci): align storybook and e2e checks
```

### Обязательные правила для коммитов

1. Название коммита должно точно соответствовать сути изменений.
2. Один коммит должен решать одну логически завершённую задачу.
3. В коммит должны попадать только файлы, которые действительно относятся к этой задаче.
4. Нельзя смешивать в одном коммите разные типы изменений без необходимости.

Плохие примеры:

- одновременно изменение компонента, тестов, документации и CI без общей причины;
- исправление линта в случайных несвязанных файлах "заодно";
- добавление инфраструктурных правок в коммит, который называется как изменение одного компонента.

Хорошая практика:

- `feat(...)` только для новой функциональности;
- `fix(...)` только для исправления дефекта;
- `docs(...)` только для документации;
- `test(...)` только для тестового контура;
- `chore(...)` для технических изменений без пользовательской функциональности.

### Требования к содержанию коммита

Каждый коммит должен быть:

1. Рабочим.
2. Самодостаточным.
3. Проверяемым.

Это означает:

1. Коммит не должен оставлять проект в заведомо сломанном состоянии.
2. После коммита код должен собираться и проходить релевантные проверки для изменённой области.
3. Нельзя коммитить промежуточное состояние, которое "починится в следующем коммите", если этого можно избежать.
4. Если изменение требует обновления тестов, stories или документации, они должны входить в тот же коммит, если относятся к одной задаче.

### Чего делать нельзя

1. Делать коммит с названием, которое не отражает фактические изменения.
2. Коммитить несвязанные правки "пакетом", чтобы сократить число коммитов.
3. Оставлять в коммите неиспользуемые файлы, временные правки, закомментированный мусор или случайные форматирования не по теме.
4. Делать коммит, если локально не пройдены обязательные проверки для этой задачи.

## Подготовка изменений перед PR

Перед созданием pull request изменения нужно привести в чистое состояние и прогнать обязательные проверки.

### Команды

Автоматическое исправление форматирования и линта:

```shell
npm run check:fix
```

Полный прогон обязательных проверок:

```shell
npm run check
```

В `npm run check` сейчас входят:

1. `npm run format:check`
2. `npm run lint`
3. `npm run typecheck`
4. `npm run test:run`
5. `npm run test:e2e`

Подробные правила по организации unit-тестов, e2e и Storybook stories вынесены в [tests/TESTING_README.md](tests/TESTING_README.md).

### Импорты

Порядок импортов поддерживается автоматически через ESLint.

Что это означает:

1. Не нужно вручную переставлять import-блоки "на глаз".
2. Если порядок импортов сбился, нужно запускать `npm run lint:fix` или `npm run check:fix`.
3. В PR не должно оставаться файлов с импортами, которые расходятся с автосортировкой линтера.

### Обязательный порядок перед PR

1. Убедиться, что изменения логически собраны и разбиты на корректные коммиты.
2. Запустить `npm run check:fix`, если нужны автоисправления форматирования и линта.
3. Запустить `npm run check`.
4. Убедиться, что PR содержит только файлы, относящиеся к задаче.
5. Проверить, что описание PR соответствует фактическим изменениям.

### Что обязательно проверить вручную

Перед PR нужно убедиться в следующем:

1. Изменение действительно решает заявленную задачу.
2. В PR нет случайных файлов и побочных правок.
3. Название PR и названия коммитов отражают фактическое содержание.
4. Все обязательные проверки завершились успешно.
5. Если менялся компонент, обновлены связанные tests, stories и документация, когда это необходимо.

Если PR затрагивает тестовую структуру или stories, нужно дополнительно проверить соответствие правилам из [tests/TESTING_README.md](tests/TESTING_README.md).

## Release и publish

Публикация пакета завязана на GitHub Release.

### Текущий релизный поток

1. Убедиться, что рабочее дерево чистое и изменения уже в `main`.
2. Локально прогнать `npm run check`.
3. Подготовить релиз через `npm run release`.
4. Проверить созданные изменения версии и changelog.
5. Запушить commit и tag командой `git push origin main --follow-tags`.
6. Создать GitHub Release по этому tag.
7. Дождаться workflow [`.github/workflows/npm_release.yaml`](.github/workflows/npm_release.yaml), который:
   - поднимается на событие `release.published`;
   - повторно запускает format, lint, unit, e2e, build;
   - проверяет tarball через `npm pack --dry-run`;
   - публикует пакет в npm только после успешной верификации.

### Что важно

1. `standard-version` по умолчанию создаёт tag с префиксом `v`, и GitHub Release должен создаваться по нему.
2. Публикация выполняется из чистого runner и всегда пересобирает `dist`, поэтому release не зависит от локально собранных файлов.
3. Если нужен повторный запуск после инфраструктурного сбоя, workflow можно перезапустить через `workflow_dispatch`, но штатный путь публикации остаётся через GitHub Release.

## Accessibility

Accessibility является обязательной частью качества компонентов, а не дополнительной опцией "на потом".

### Базовое правило

Все новые stories должны проходить проверки `@storybook/addon-a11y`.

В проекте Storybook accessibility включён в строгом режиме. Это означает:

1. Accessibility violations в stories считаются ошибкой.
2. Новый компонент нельзя считать готовым, если его story падает по a11y.
3. Исправление accessibility не должно откладываться в отдельную задачу без явной причины.

### Что проверяется автоматически

Storybook accessibility checks помогают ловить, в частности:

1. Проблемы с ролями и семантикой.
2. Отсутствующие или некорректные label/aria-атрибуты.
3. Частые ошибки структуры markup.
4. Часть проблем с визуальной доступностью и контрастом.

### Что остаётся обязанностью автора компонента

Автоматические проверки не заменяют инженерное решение. Автор компонента обязан проверить:

1. Корректную семантику компонента.
2. Работу с клавиатуры, если компонент интерактивный.
3. Видимый focus-state.
4. Корректное поведение disabled/readOnly/error/opened/selected состояний, если они есть у компонента.
5. Понятный и безопасный API для использования компонента без скрытых accessibility-требований.

### Минимальный accessibility DoD

Для любого нового компонента минимум включает:

1. Есть story в Storybook.
2. Story проходит Storybook accessibility checks без violations.
3. Если компонент интерактивный, поведение с клавиатуры и focus покрыто тестом или явно проверено вручную.
4. Если для доступного использования от потребителя требуется обязательный prop или связка prop-ов, это должно быть отражено в документации и API компонента.

## Новый компонент

Новый компонент должен добавляться по единому шаблону. Эталонный пример в текущем репозитории: `ExampleComponent`.

### Обязательная структура

Минимальная структура нового компонента:

```text
src/components/ComponentName/
  ComponentName.tsx
  ComponentName.test.tsx
  index.ts
  stories/
    ComponentName.stories.tsx
    ComponentNamePlayground.template.tsx
tests/e2e/ComponentName/
  component-name.spec.ts
```

Допустимые дополнительные файлы:

1. `style.ts`
2. `constants.ts`
3. `types.ts`
4. вспомогательные файлы рядом с компонентом, если без них нельзя сохранить читаемую структуру

### Назначение файлов

1. `ComponentName.tsx`
   Основная реализация компонента и его props.
2. `index.ts`
   Локальная точка экспорта компонента и его публичных типов.
3. `ComponentName.test.tsx`
   Unit-тесты компонента.
4. `ComponentName.stories.tsx`
   Входной Storybook CSF-файл с `meta` и экспортами stories.
5. `ComponentNamePlayground.template.tsx`
   React-шаблоны и render-функции без Storybook-специфики. Может переиспользоваться и в Storybook, и в internal playground.
6. `tests/e2e/ComponentName/*.spec.ts`
   Smoke/e2e-проверки internal playground для компонента.

Если у компонента нужны отдельные внутренние templates, которые не должны быть видны в Storybook, их можно хранить рядом в `stories/`, но не импортировать в `*.stories.tsx`.

Пример:

```text
src/components/ComponentName/stories/ComponentNameDirty.template.tsx
```

### Правила для Storybook

1. `*.stories.tsx` должен оставаться тонким.
2. Storybook-специфика (`Meta`, `StoryObj`, `args`, `argTypes`, story exports) должна находиться в `*.stories.tsx`.
3. `*.template.tsx` должен содержать только React-код и не должен зависеть от типов или объектов Storybook.
4. Stories должны использовать публичный импорт пакета, то есть так, как компонент будет использоваться потребителем.
5. Не каждый `*.template.tsx` обязан попадать в Storybook. Internal-only templates допустимы, если они используются только в playground/e2e.

### Правила для internal playground

1. E2E не должны зависеть от Storybook runtime.
2. Сценарии playground должны описываться в `playground/scenarios`.
3. `playground/scenarios/index.ts` должен оставаться агрегатором, а не большим монолитным списком всех сценариев вручную.
4. Для нового компонента или группы кейсов нужно добавлять отдельный файл сценариев, например `playground/scenarios/component-name.tsx`.
5. В сценариях playground можно использовать как публичные story templates, так и internal-only templates, если это оправдано тестовым кейсом.

### Правила для API и экспортов

1. Публичный API пакета собирается только через `src/index.ts`.
2. Новый компонент обязан быть добавлен в `src/index.ts`.
3. Внутри папки компонента должен существовать локальный `index.ts`.
4. Корневой `src/index.ts` должен реэкспортировать компонент через `export * from './components/ComponentName';`.
5. Источником публичного API компонента является локальный `index.ts`, а не прямой импорт файла реализации.
6. Deep imports не считаются частью публичного API.
7. Story-only args не должны подменять реальные props компонента, если это не оправдано отдельным кейсом.
8. Если story использует дополнительные args для удобства playground, это должно быть осознанным исключением, а не базовым паттерном.

### Правила для props и нейминга

1. Props компонента должны быть объявлены явным интерфейсом.
2. Публичные props и ключевые типы должны иметь понятный нейминг без внутренних сокращений.
3. Если prop влияет на доступность или корректность использования, это должно быть видно по API и документации.
4. Для публичных props и типов желательно добавлять JSDoc, чтобы Autodocs показывал полезные описания.

### Минимальный Definition of Done для нового компонента

Компонент считается готовым к PR, когда выполняются все условия:

1. Компонент реализован в собственной папке в `src/components`.
2. Компонент экспортируется через локальный `index.ts` и через `src/index.ts`.
3. Для компонента есть хотя бы одна Storybook story.
4. Story использует `autodocs`.
5. Story проходит Storybook accessibility checks.
6. Есть unit-тесты на базовое поведение компонента.
7. Есть e2e/smoke-проверка через internal playground.
8. Для интерактивного компонента проверены keyboard/focus-сценарии.
9. Публичный API компонента не требует скрытых договорённостей.
10. Все локальные проверки для изменённой области проходят.

## Минимальное ожидание от каждого PR

Каждый PR должен быть:

1. Ограничен одной задачей или одной группой тесно связанных изменений.
2. Читаем по истории коммитов.
3. Проверяем локально.
4. Готов к ревью без необходимости "доделать потом".

PR не должен использоваться как контейнер для незавершённой или грязной работы, если это не оговорено отдельно.
