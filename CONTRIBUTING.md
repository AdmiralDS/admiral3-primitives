# Contributing

В этом документе зафиксированы обязательные правила внесения изменений в репозиторий.

## Быстрый старт для разработки

Этот раздел помогает быстро начать работу в проекте. Подробные правила и пояснения находятся ниже в документе.

### Первый запуск

1. Установить актуальную Node.js LTS. CI использует Node.js 24.
2. Установить зависимости:

```shell
npm ci
```

3. Проверить проект:

```shell
npm run check:full
```

Если проверка упала, сначала разберите ошибку и не начинайте изменения поверх сломанного состояния.

### Обычный рабочий цикл

1. Взять одну небольшую задачу.
2. Внести минимальные изменения по задаче.
3. Если менялся компонент, обновить связанные tests, stories, playground/e2e сценарии и документацию, когда это нужно.
4. Запустить автоисправления форматирования и линта:

```shell
npm run check:fix
```

5. Запустить полный набор проверок:

```shell
npm run check:full
```

6. Перед PR ещё раз проверить `git status` и убедиться, что в изменения попали только файлы по задаче.

### Если нужно добавить компонент

1. Сгенерировать базовую структуру:

```shell
npm run generate:component -- ComponentName
```

2. Заменить placeholder-реализацию на реальный API компонента.
3. Проверить локальный export в `src/components/ComponentName/index.ts` и root export в `src/index.ts`.
4. Добавить или уточнить unit-тесты, Storybook stories, playground scenario и e2e-проверку.
5. Запустить `npm run check:full`.

### Самые важные правила

1. Не добавляйте unrelated changes "заодно".
2. Не используйте inline-стили `style={{ ... }}` для постоянной разметки компонентов, stories и templates.
3. Типографику берите готовым объектом из токенов, например `textStyles.body.body2Long`.
4. Публичные props и типы компонента экспортируйте через локальный `index.ts` и root `src/index.ts`.
5. Импортируйте компоненты в stories/templates как потребитель: из `@admiral-ds/admiral3-primitives`.
6. Не добавляйте deep exports в `package.json`.
7. Не коммитьте изменения, если обязательные проверки не проходили или падали.

### Где искать детали

1. Коммиты и PR: разделы [Коммиты](#коммиты), [Подготовка изменений перед PR](#подготовка-изменений-перед-pr) и [Минимальное ожидание от каждого PR](#минимальное-ожидание-от-каждого-pr).
2. Стили и типографика: раздел [Стили и типографика](#стили-и-типографика).
3. Accessibility: раздел [Accessibility](#accessibility).
4. Новый компонент: раздел [Новый компонент](#новый-компонент).
5. Unit/e2e/Storybook правила: [tests/TESTING_README.md](tests/TESTING_README.md).

## Коммиты

Для сообщений коммитов используется `Conventional Commits`:

https://www.conventionalcommits.org/ru/v1.0.0/

### Формат сообщения

Базовый формат:

```text
type(scope): short description
```

Примеры:

```text
feat(BadgeDot): add storybook playground template
fix(testing): correct vitest typings for jest-dom matchers
docs(contributing): describe pull request preparation flow
chore(ci): align storybook and e2e checks
```

### Обязательные правила для коммитов

1. Название коммита должно точно соответствовать сути изменений.
2. Один коммит должен решать одну логически завершённую задачу.
3. В коммит должны попадать только файлы, которые действительно относятся к этой задаче.
4. Нельзя смешивать в одном коммите разные типы изменений без необходимости.

Плохие примеры:

- одновременно изменение компонента, тестов, документации и CI без общей причины;
- исправление линта в случайных несвязанных файлах "заодно";
- добавление инфраструктурных правок в коммит, который называется как изменение одного компонента.

Хорошая практика:

- `feat(...)` только для новой функциональности;
- `fix(...)` только для исправления дефекта;
- `docs(...)` только для документации;
- `test(...)` только для тестового контура;
- `chore(...)` для технических изменений без пользовательской функциональности.

### Требования к содержанию коммита

Каждый коммит должен быть:

1. Рабочим.
2. Самодостаточным.
3. Проверяемым.

Это означает:

1. Коммит не должен оставлять проект в заведомо сломанном состоянии.
2. После коммита код должен собираться и проходить релевантные проверки для изменённой области.
3. Нельзя коммитить промежуточное состояние, которое "починится в следующем коммите", если этого можно избежать.
4. Если изменение требует обновления тестов, stories или документации, они должны входить в тот же коммит, если относятся к одной задаче.

### Чего делать нельзя

1. Делать коммит с названием, которое не отражает фактические изменения.
2. Коммитить несвязанные правки "пакетом", чтобы сократить число коммитов.
3. Оставлять в коммите неиспользуемые файлы, временные правки, закомментированный мусор или случайные форматирования не по теме.
4. Делать коммит, если локально не пройдены обязательные проверки для этой задачи.

## Подготовка изменений перед PR

Перед созданием pull request изменения нужно привести в чистое состояние и прогнать обязательные проверки.

CI запускается на актуальной Node.js LTS. На момент проверки репозитория это Node.js 24.
Для локальной проверки используйте ту же major-версию Node.js и устанавливайте зависимости через `npm ci`.

### Команды

Основные локальные потоки:

```shell
npm run check:fix
npm run check:full
npm run storybook
npm run playground
npm run release
```

Автоматическое исправление форматирования и линта:

```shell
npm run check:fix
```

Полный прогон обязательных проверок:

```shell
npm run check:full
```

В `npm run check:full` сейчас входят:

1. `npm run format:check`
2. `npm run lint`
3. `npm run validate:components`
4. `npm run typecheck`
5. `npm run test:run`
6. `npm run test:e2e`

`npm run storybook` запускает Storybook из исходников и не требует предварительной сборки CSS.
`npm run playground` запускает Vite playground с hot reload. Playground может импортировать CSS из `@admiral-ds/admiral3-tokens`, но primitives-пакет не генерирует CSS при сборке.

Подробные правила по организации unit-тестов, e2e и Storybook stories вынесены в [tests/TESTING_README.md](tests/TESTING_README.md).

### Импорты

Порядок импортов поддерживается автоматически через ESLint.

Что это означает:

1. Не нужно вручную переставлять import-блоки "на глаз".
2. Если порядок импортов сбился, нужно запускать `npm run lint:fix` или `npm run check:fix`.
3. В PR не должно оставаться файлов с импортами, которые расходятся с автосортировкой линтера.

### Стили и типографика

Стили компонентов, stories и templates должны быть оформлены через `styled-components` или локальные style helpers. Inline-стили через `style={{ ... }}` не используются для постоянной разметки компонента или story.

Размерные параметры компонентов, которые могут использоваться в расчётах, нужно хранить числовыми значениями без CSS-единиц. Единицы измерения, например `px`, добавляются на уровне CSS в `styled-components`.

Если для компонента нужен список допустимых значений, например `appearance` или `dimension`, список должен быть единственным источником истины. Храните его в constants через `as const`, а TypeScript union выводите из этого списка через `(typeof SOME_VALUES)[number]`.

Если компонент поддерживает preset-цвета и пользовательские цвета, держите API явным: `Appearance | ColorConfig`. Пользовательский config должен принимать CSS-значения или CSS custom properties, а реализация должна отделять preset-ветку от custom-ветки без передачи object-значений в DOM.

Если story показывает пользовательские цвета для компонента с текстом на цветном фоне, добавляйте пояснение о необходимости подбирать читаемую пару цветов фона и текста.

Типографика всегда применяется через готовый объект стиля из токенов, например `textStyles.body.body2Long`. Нельзя вручную разбивать типографику на отдельные свойства `font-family`, `font-weight`, `font-size`, `line-height`, если в токенах уже есть соответствующий объект стиля.

### Обязательный порядок перед PR

1. Убедиться, что изменения логически собраны и разбиты на корректные коммиты.
2. Запустить `npm run check:fix`, если нужны автоисправления форматирования и линта.
3. Запустить `npm run check:full`.
4. Убедиться, что PR содержит только файлы, относящиеся к задаче.
5. Проверить, что описание PR соответствует фактическим изменениям.

### Что обязательно проверить вручную

Перед PR нужно убедиться в следующем:

1. Изменение действительно решает заявленную задачу.
2. В PR нет случайных файлов и побочных правок.
3. Название PR и названия коммитов отражают фактическое содержание.
4. Все обязательные проверки завершились успешно.
5. Если менялся компонент, обновлены связанные tests, stories и документация, когда это необходимо.

Если PR затрагивает тестовую структуру или stories, нужно дополнительно проверить соответствие правилам из [tests/TESTING_README.md](tests/TESTING_README.md).

### README и публичный API

README описывает установку, peer dependencies и поддерживаемый root entry point пакета. Он не является каталогом всех
компонентов и не должен обновляться только потому, что в `src/index.ts` добавлен очередной компонент.

Обновляйте README, если меняется способ установки, набор peer dependencies, общий способ использования пакета, root
entry point или другая информация, важная внешнему пользователю библиотеки. Для правил разработки, структуры компонентов,
тестов и внутренних сценариев используйте CONTRIBUTING, PROJECT-MAP и tests/TESTING_README.

## Release и publish

Публикация пакета завязана на GitHub Release.

### Текущий релизный поток

1. Убедиться, что рабочее дерево чистое и изменения уже в `main`.
2. Локально прогнать `npm run check:full`.
3. Подготовить релиз через `npm run release`.
4. Проверить созданные изменения версии и changelog.
5. Запушить commit и tag командой `git push origin main --follow-tags`.
6. Создать GitHub Release по этому tag.
7. Дождаться workflow [`.github/workflows/npm_release.yaml`](.github/workflows/npm_release.yaml), который:
   - поднимается на событие `release.published`;
   - повторно запускает полный обязательный контур `npm run check:full` и build;
   - проверяет tarball через `npm run pack:check`: dry-run состава npm-пакета и lightweight size check для `dist/index.js`;
   - публикует пакет в npm только после успешной верификации.

### Что важно

1. `standard-version` по умолчанию создаёт tag с префиксом `v`, и GitHub Release должен создаваться по нему.
2. Публикация выполняется из чистого runner и всегда пересобирает `dist`, поэтому release не зависит от локально собранных файлов.
3. Если нужен повторный запуск после инфраструктурного сбоя, workflow можно перезапустить через `workflow_dispatch`, но штатный путь публикации остаётся через GitHub Release.

## Accessibility

Accessibility является обязательной частью качества компонентов, а не дополнительной опцией "на потом".

### Базовое правило

Все новые stories должны проходить проверки `@storybook/addon-a11y`.

В проекте Storybook accessibility включён в строгом режиме. Это означает:

1. Accessibility violations в stories считаются ошибкой.
2. Новый компонент нельзя считать готовым, если его story падает по a11y.
3. Исправление accessibility не должно откладываться в отдельную задачу без явной причины.

### Что проверяется автоматически

Storybook accessibility checks помогают ловить, в частности:

1. Проблемы с ролями и семантикой.
2. Отсутствующие или некорректные label/aria-атрибуты.
3. Частые ошибки структуры markup.
4. Часть проблем с визуальной доступностью и контрастом.

### Что остаётся обязанностью автора компонента

Автоматические проверки не заменяют инженерное решение. Автор компонента обязан проверить:

1. Корректную семантику компонента.
2. Работу с клавиатуры, если компонент интерактивный.
3. Видимый focus-state.
4. Корректное поведение disabled/readOnly/error/opened/selected состояний, если они есть у компонента.
5. Понятный и безопасный API для использования компонента без скрытых accessibility-требований.

### Минимальный accessibility DoD

Для любого нового компонента минимум включает:

1. Есть story в Storybook.
2. Story проходит Storybook accessibility checks без violations.
3. Если компонент интерактивный, поведение с клавиатуры и focus покрыто тестом или явно проверено вручную.
4. Если для доступного использования от потребителя требуется обязательный prop или связка prop-ов, это должно быть отражено в документации и API компонента.

## Новый компонент

Новый компонент должен добавляться по единому шаблону. Эталонный пример в текущем репозитории: `BadgeDot`.

Для базового scaffolding используется `generate-react-cli` с проектными templates. Команда:

```shell
npm run generate:component -- ComponentName
```

Если команда запускается без аргумента из интерактивного терминала, например из панели npm scripts в VS Code, скрипт попросит ввести имя компонента. Команда принимает имя компонента в `PascalCase` и создает:

1. каталог `src/components/ComponentName`;
2. файлы компонента `ComponentName.tsx`, `style.ts`, `types.ts`, `constants.ts`, `index.ts` и `ComponentName.test.tsx`;
3. Storybook-файлы `ComponentName.stories.tsx` и `ComponentNamePlayground.template.tsx`;
4. playground-сценарий `playground/scenarios/component-name.tsx`;
5. e2e smoke-тест `tests/e2e/ComponentName/component-name.spec.ts`;
6. экспорт компонента в `src/index.ts`;
7. подключение сценария в `playground/scenarios/index.ts`.

Конфигурация находится в `generate-react-cli.json`, templates - в `scripts/templates/generate-react-component`.
Сгенерированный код является стартовым шаблоном. После генерации нужно заменить placeholder-реализацию на фактический API компонента, расширить stories/tests/e2e под реальные состояния и запустить обязательные проверки.

Структурные правила компонента проверяются командой:

```shell
npm run validate:components
```

Эта команда входит в `npm run check:full` и проверяет наличие обязательных файлов, root export, отсутствие deep exports в публичном API, экспорт компонента и публичных props/types из локального `index.ts`, связь playground-сценария с e2e-спекой и отсутствие импортов публичных props/types из внутренних файлов в `*.stories.tsx` и `*.template.tsx`.

### Обязательная структура

Минимальная структура нового компонента:

```text
src/components/ComponentName/
  ComponentName.tsx
  ComponentName.test.tsx
  index.ts
  types.ts
  stories/
    ComponentName.stories.tsx
    ComponentNamePlayground.template.tsx
tests/e2e/ComponentName/
  component-name.spec.ts
```

Допустимые дополнительные файлы:

1. `style.ts`
2. `constants.ts`
3. вспомогательные файлы рядом с компонентом, если без них нельзя сохранить читаемую структуру

### Назначение файлов

1. `ComponentName.tsx`
   Основная реализация компонента. Публичные props импортируются из `types.ts`.
2. `index.ts`
   Локальная точка экспорта компонента и его публичных типов.
3. `types.ts`
   Типы компонента: публичные props, публичные union-типы и внутренние служебные типы, если они нужны только внутри папки компонента.
4. `ComponentName.test.tsx`
   Unit-тесты компонента.
5. `ComponentName.stories.tsx`
   Входной Storybook CSF-файл с `meta` и экспортами stories.
6. `ComponentNamePlayground.template.tsx`
   React-шаблоны и render-функции без Storybook-специфики. Может переиспользоваться и в Storybook, и в internal playground.
7. `tests/e2e/ComponentName/*.spec.ts`
   Smoke/e2e-проверки internal playground для компонента.

Если у компонента нужны отдельные внутренние templates, которые не должны быть видны в Storybook, их можно хранить рядом в `stories/`, но не импортировать в `*.stories.tsx`.

Пример:

```text
src/components/ComponentName/stories/ComponentNameDirty.template.tsx
```

### Правила для Storybook

1. `*.stories.tsx` должен оставаться тонким.
2. Storybook-специфика (`Meta`, `StoryObj`, `args`, `argTypes`, story exports) должна находиться в `*.stories.tsx`.
3. `*.template.tsx` должен содержать только React-код и не должен зависеть от типов или объектов Storybook.
4. Stories должны использовать публичный импорт пакета, то есть так, как компонент будет использоваться потребителем.
5. `*.stories.tsx` и `*.template.tsx` должны импортировать публичные компоненты, props и публичные типы из `@admiral-ds/admiral3-primitives`. Локальные импорты из папки компонента допустимы только для внутренних служебных сущностей, которые не нужны финальному пользователю библиотеки, например компонентных `constants`.
6. Не каждый `*.template.tsx` обязан попадать в Storybook. Internal-only templates допустимы, если они используются только в playground/e2e.
7. Общие контейнеры и layout helpers для templates хранятся в `src/components/stories`. Используйте их, если layout должен одинаково работать в Storybook и internal playground.
8. Глобальные decorators и styles в `.storybook` используются только для Storybook shell: theme, fonts, docs/canvas padding и базовое выравнивание. Layout, который является частью demo-кейса или e2e-сценария, должен оставаться внутри template/helper.

### Правила для internal playground

1. E2E не должны зависеть от Storybook runtime.
2. Сценарии playground должны описываться в `playground/scenarios`.
3. `playground/scenarios/index.ts` должен оставаться агрегатором, а не большим монолитным списком всех сценариев вручную.
4. Для нового компонента или группы кейсов нужно добавлять отдельный файл сценариев, например `playground/scenarios/component-name.tsx`.
5. В сценариях playground можно использовать как публичные story templates, так и internal-only templates, если это оправдано тестовым кейсом.

### Правила для API и экспортов

1. Публичный API пакета собирается только через `src/index.ts`.
2. Новый компонент обязан быть добавлен в `src/index.ts`.
3. Внутри папки компонента должен существовать локальный `index.ts`.
4. Корневой `src/index.ts` должен реэкспортировать компонент через `export * from './components/ComponentName';`.
5. Источником публичного API компонента является локальный `index.ts`, а не прямой импорт файла реализации.
6. Deep imports не считаются частью публичного API.
7. Story-only args не должны подменять реальные props компонента, если это не оправдано отдельным кейсом.
8. Если story использует дополнительные args для удобства playground, это должно быть осознанным исключением, а не базовым паттерном.
9. Публичные props компонента и публичные union-типы должны объявляться в локальном `types.ts`.
10. Локальный `index.ts` компонента должен экспортировать только сам компонент named-экспортом, его props и публичные типы. Внутренние `constants.ts`, `style.ts`, styled props и служебные helpers не экспортируются наружу.
11. В `package.json` не добавляются deep exports для компонентных файлов или папок; публичным entrypoint остаётся корень пакета.

### Правила для props и нейминга

1. Props компонента должны быть объявлены явным интерфейсом.
2. Интерфейс props должен лежать в `types.ts`, даже если пока он короткий. Это сохраняет единый шаблон для будущих сложных компонентов.
3. В одном `types.ts` допустимо хранить публичные и внутренние типы компонента. Наружу через `index.ts` экспортируются только публичные типы.
4. Публичные props и ключевые типы должны иметь понятный нейминг без внутренних сокращений.
5. Если prop влияет на доступность или корректность использования, это должно быть видно по API и документации.
6. Для публичных props и типов желательно добавлять JSDoc, чтобы Autodocs показывал полезные описания.

### Минимальный Definition of Done для нового компонента

Компонент считается готовым к PR, когда выполняются все условия:

1. Компонент реализован в собственной папке в `src/components`.
2. Компонент экспортируется через локальный `index.ts` и через `src/index.ts`.
3. Для компонента есть хотя бы одна Storybook story.
4. Story использует `autodocs`.
5. Story проходит Storybook accessibility checks.
6. Есть unit-тесты на базовое поведение компонента.
7. Есть e2e/smoke-проверка через internal playground.
8. Для интерактивного компонента проверены keyboard/focus-сценарии.
9. Публичный API компонента не требует скрытых договорённостей.
10. Все локальные проверки для изменённой области проходят.

## Минимальное ожидание от каждого PR

Каждый PR должен быть:

1. Ограничен одной задачей или одной группой тесно связанных изменений.
2. Читаем по истории коммитов.
3. Проверяем локально.
4. Готов к ревью без необходимости "доделать потом".

PR не должен использоваться как контейнер для незавершённой или грязной работы, если это не оговорено отдельно.
